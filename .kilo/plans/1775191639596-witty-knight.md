## Issues Found

### 1. Medium: `move_player` Still Accepts `routeEdgeId` in Tests But Not in Type

The `MechanicsMutation` type for `move_player` has `routeEdgeId?: string` (optional), but the provider Zod schema still has `routeEdgeId: z.string().trim().min(1).optional()`. The engine's `move_player` handler in `evaluateResolvedCommand` does NOT validate `routeEdgeId` at all — it only validates `targetLocationId` exists as a `LocationNode`. This means the `routeEdgeId` field is dead weight in the mutation.

**Impact:** Low. It's a vestigial field from the old travel model. The tests that pass `routeEdgeId` with `move_player` still work because the field is optional and ignored.

**Recommendation:** Remove `routeEdgeId` from the `move_player` mutation entirely. It serves no purpose for magical relocation.

### 2. Medium: `buildDiscoveryProjection` Doesn't Filter `candidateLocations` by `discoveryState`

In `buildDiscoveryProjection`, `candidateLocations` queries locations where `discoveryState === "ambient"`. But the spec says `latentTargets` should include "nearby hidden routes/locations not currently eligible." The current implementation includes ambient locations in the candidate set, which means they appear as latent targets even though they're truly undiscovered (ambient = not yet hinted at).

**Impact:** This is actually correct per the spec — latent targets are meant to be discoverable through `force_reveal_discovery` even if not yet revealed by information. The "ambient" state means the location exists but the player hasn't heard of it yet. The latent target list gives the LLM a legal alias to reveal it through creative play.

**Verdict:** Not a bug, but worth noting that the game design implication is that any nearby ambient location can be discovered through `force_reveal_discovery` without any information prerequisite.

### 3. Low: `CampaignPromptCache` Upsert Race Condition

The `getPromptContext` function has this guard:
```typescript
if (!promptRequestId || existingPromptCache?.promptRequestId === promptRequestId) {
  await prisma.campaignPromptCache.upsert({ ... });
}
```

This means if two concurrent requests generate prompt context for the same campaign, the second one will overwrite the first one's `promptRequestId` if they both pass the guard. Since `promptRequestId` is generated fresh each time (`randomUUID()`), this could cause a stale context error for the first request's turn submission.

**Impact:** Low in practice — the NDJSON streaming model means only one turn is active at a time. But if two tabs are open, this could cause issues.

**Recommendation:** Add a Prisma-level unique constraint check or optimistic concurrency control. The current single-row model is inherently racy under concurrent access.

### 4. Low: `toRouteSummary` Uses `isKnown` from `knownRouteIds` But Doesn't Account for Public Routes

The `toRouteSummary` function receives `isKnown: knownRouteIds.has(edge.id)`. But public routes don't have `CampaignKnownRoute` rows — they're auto-surfaced by the visibility filter. So `isKnown` will be `false` for public routes.

**Impact:** Low. The `isKnown` field is used to distinguish "learned through discovery" from "always visible." Public routes being `isKnown: false` is semantically correct — the player hasn't "discovered" them, they've always been visible.

### 5. Low: Missing Causality Code Registration

The spec says "distinct causality/reason codes" for `resolve_discovery_hook` vs `force_reveal_discovery`. The engine uses:
- `authored_discovery_requested` / `fallback_discovery_requested` as `reasonCode`
- `authored_discovery_revealed` / `fallback_discovery_revealed` as the commit log reason code

But these are NOT registered in the `turnCausalityCodeNameValues` array in `json-contracts.ts`. The causality code schema is an enum, and these new codes are not in it.

**Impact:** This will cause Zod validation failures when the turn result payload is serialized/deserialized. The `turnCausalityCodeSchema` in `json-contracts.ts` has a fixed enum of allowed codes.

**Recommendation:** Add the new causality codes to `turnCausalityCodeNameValues` in `json-contracts.ts`:
- `AUTHORED_DISCOVERY_REVEALED`
- `FALLBACK_DISCOVERY_REVEALED`
- `JOURNEY_STARTED`
- `JOURNEY_ARRIVED`
- `JOURNEY_ABORTED`
- `JOURNEY_REVERSED`

### 6. Low: `world-persistence-translation.ts` Not Integrated Into Launch Flow

The `buildInformationRevealRows` function exists but I don't see it called in `prepareCampaignLaunch` or the campaign creation flow. The spec says "Add a separate persistence translation layer that converts nested reveal lists." This function builds the rows but needs to be called during campaign initialization.

**Impact:** This could mean information reveal rows are never actually created during campaign launch. The generation produces `revealsEdgeIds`/`revealsLocationIds`, the translation function can convert them, but if nobody calls it, the rows never get inserted.

**Recommendation:** Verify that `buildInformationRevealRows` is called during `prepareCampaignLaunch` and the resulting rows are inserted via Prisma `createMany`.

### 7. Low: `adjacentRoutes` Returns Empty Array During Journeys

When `currentLocationRecord` is null (during a journey), `adjacentRoutes` is set to `[]`. This is correct — the player isn't at a location, so there are no adjacent routes. But the UI still renders the "Routes and Leads" section with empty content during journeys.

**Impact:** The journey UI replaces the route section, so this is fine. But if the journey UI fails to render for some reason, the player would see an empty routes section.

## Test Coverage Gaps

The existing tests were updated for the new mutation shapes, but several spec requirements are not covered by tests:

1. **`CampaignPromptCache` validation** — no test for stale `promptRequestId` rejection
2. **`force_reveal_discovery` vs `resolve_discovery_hook`** — no test for distinct causality codes
3. **`locationLeads` computation** — no test for parent-region scoping
4. **Journey teardown during magical relocation** — no test for the snap-to-nearer-endpoint logic
5. **`buildInformationRevealRows`** — no test for the persistence translation layer
6. **`adjacentRoutes` visibility filter** — no test for `visibility === "public" OR CampaignKnownRoute`
7. **Entry point validation** — no test for "start location cannot be ambient"
